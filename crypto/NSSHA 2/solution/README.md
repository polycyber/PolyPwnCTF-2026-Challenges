# NSSHA (2)

If we look at the source code for the second part, we can see that it is practically the same as the first part, however, with the added feature that we can hash our own messages.

If we perform our previous operations to determine Python's hashing configuration, we can see that it still uses FNV, but with a key.

Unfortunately for us, this means the key is [generated by a CSPRNG](https://github.com/python/cpython/blob/5fa025aebe8d47a9ca56d5acbb7bb80898809de0/Python/bootstrap_hash.c#L582)

The way to solve this though, is by simply adding more constraints to our solver.

In our solver, we can create two new variables for the prefix and suffix keys, and add more constraints that define known outputs (hashes) for known inputs (strings).

By doing this, we can narrow down the prefix and suffix to a few values. During my testing there were 2 candidate pairs.

After that, we apply the same logic as the first solution, but by applying the keys in their correct position.

The key here though is to reuse the same variables (prefix, suffix) across all FNV operations. This way, Z3 can actually solve them based on all the constraints.

The full code can be found [here](./fnv_keyed.py).

In this case, we get a few collisions. By just reading them though, we can find the correct one as it most resembles English:

